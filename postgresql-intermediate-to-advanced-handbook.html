Module 1: Advanced Joins & Set Operators

CROSS JOIN

A cross join produces every combination. It is the Cartesian product of two sets. That sounds dangerous, and it can be, but it is useful when you intentionally want all combinations, such as generating a price grid across regions and shipping tiers.

SELECT
    region.region_name,
    tier.tier_name
FROM (VALUES ('North America'), ('Europe'), ('APAC')) AS region(region_name)
CROSS JOIN (VALUES ('Standard'), ('Express')) AS tier(tier_name)
ORDER BY region.region_name, tier.tier_name;

LATERAL

LATERAL behaves like a for-each loop in SQL. For each row from the left side, PostgreSQL runs the lateral subquery using values from that row. It is perfect when you need the top related row, a derived slice, or a parameterized subquery without writing procedural code.

SELECT
    u.user_id,
    u.full_name,
    latest_order.order_id,
    latest_order.order_date,
    latest_order.total_amount
FROM sales.users AS u
LEFT JOIN LATERAL (
    SELECT o.order_id, o.order_date, o.total_amount
    FROM sales.orders AS o
    WHERE o.user_id = u.user_id
    ORDER BY o.order_date DESC
    LIMIT 1
) AS latest_order
    ON TRUE
ORDER BY u.user_id;

Set Operators

Set operators combine result sets. Use them when the question is about relationships between result groups instead of row-by-row joins.

SELECT email
FROM sales.marketing_subscribers
UNION
SELECT email
FROM sales.users;

UNION removes duplicates. That extra deduplication costs work. If duplicates are acceptable or desirable, UNION ALL is faster because PostgreSQL can concatenate results directly.

SELECT email
FROM sales.marketing_subscribers
UNION ALL
SELECT email
FROM sales.users;
SELECT email
FROM sales.marketing_subscribers
INTERSECT
SELECT email
FROM sales.users;
SELECT email
FROM sales.marketing_subscribers
EXCEPT
SELECT email
FROM sales.users;

Module 2: Window Functions & Ranking

The OVER() clause defines the analytical window. Unlike GROUP BY, which collapses many rows into one summary row, a window function keeps each original row and adds a calculation beside it.

Ranking Functions

ROW_NUMBER() gives a unique sequence. RANK() leaves gaps after ties. DENSE_RANK() keeps ranking compact even when ties occur.

SELECT
    user_id,
    order_id,
    total_amount,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY total_amount DESC) AS row_number_rank,
    RANK() OVER (PARTITION BY user_id ORDER BY total_amount DESC) AS rank_value,
    DENSE_RANK() OVER (PARTITION BY user_id ORDER BY total_amount DESC) AS dense_rank_value
FROM sales.orders
ORDER BY user_id, total_amount DESC;

LAG() and LEAD()

These functions let one row look at its neighbors. That is useful for change analysis such as week-over-week growth, trend detection, or comparing a shipment to the previous shipment for the same warehouse.

WITH weekly_sales AS (
    SELECT
        DATE_TRUNC('week', order_date) AS sales_week,
        SUM(total_amount) AS weekly_revenue
    FROM sales.orders
    WHERE order_status IN ('paid', 'shipped', 'delivered')
    GROUP BY DATE_TRUNC('week', order_date)
)
SELECT
    sales_week,
    weekly_revenue,
    LAG(weekly_revenue) OVER (ORDER BY sales_week) AS previous_week_revenue,
    LEAD(weekly_revenue) OVER (ORDER BY sales_week) AS next_week_revenue,
    weekly_revenue - LAG(weekly_revenue) OVER (ORDER BY sales_week) AS week_over_week_change
FROM weekly_sales
ORDER BY sales_week;

Module 3: Common Table Expressions (CTEs)

Chaining Standard CTEs

Multiple CTEs let you break a large query into named steps. That makes logic review much easier than tracing through deeply nested subqueries.

WITH paid_orders AS (
    SELECT user_id, total_amount, order_date
    FROM sales.orders
    WHERE order_status IN ('paid', 'shipped', 'delivered')
),
user_order_totals AS (
    SELECT
        user_id,
        COUNT(*) AS completed_order_count,
        SUM(total_amount) AS lifetime_value,
        MAX(order_date) AS last_order_date
    FROM paid_orders
    GROUP BY user_id
),
vip_users AS (
    SELECT *
    FROM user_order_totals
    WHERE lifetime_value >= 5000.00
)
SELECT
    u.user_id,
    u.full_name,
    v.completed_order_count,
    v.lifetime_value,
    v.last_order_date
FROM vip_users AS v
INNER JOIN sales.users AS u
    ON u.user_id = v.user_id
ORDER BY v.lifetime_value DESC;

Recursive CTEs

WITH RECURSIVE traverses hierarchical data. You define an anchor query for the starting rows and a recursive member that walks to the next level. This is ideal for org charts, category trees, or bills of materials.

CREATE TABLE hr.employees (
    employee_id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    employee_name TEXT NOT NULL,
    manager_id INTEGER REFERENCES hr.employees(employee_id)
);
WITH RECURSIVE org_chart AS (
    SELECT
        employee_id,
        employee_name,
        manager_id,
        0 AS depth,
        employee_name::TEXT AS reporting_path
    FROM hr.employees
    WHERE manager_id IS NULL

    UNION ALL

    SELECT
        e.employee_id,
        e.employee_name,
        e.manager_id,
        oc.depth + 1 AS depth,
        oc.reporting_path || ' -> ' || e.employee_name AS reporting_path
    FROM hr.employees AS e
    INNER JOIN org_chart AS oc
        ON e.manager_id = oc.employee_id
)
SELECT employee_id, employee_name, manager_id, depth, reporting_path
FROM org_chart
ORDER BY reporting_path;

Module 4: Advanced Grouping & Pivoting

ROLLUP, CUBE, and GROUPING SETS

Advanced grouping creates subtotals and grand totals in one pass. This is extremely useful in finance and operations reporting where multiple levels of aggregation are expected from the same dataset.

SELECT
    sales_region,
    order_status,
    SUM(total_amount) AS revenue
FROM sales.orders
GROUP BY ROLLUP (sales_region, order_status)
ORDER BY sales_region, order_status;
SELECT
    sales_region,
    order_status,
    SUM(total_amount) AS revenue
FROM sales.orders
GROUP BY CUBE (sales_region, order_status)
ORDER BY sales_region, order_status;
SELECT
    sales_region,
    payment_method,
    SUM(total_amount) AS revenue
FROM sales.orders
GROUP BY GROUPING SETS (
    (sales_region, payment_method),
    (sales_region),
    (payment_method),
    ()
)
ORDER BY sales_region, payment_method;

Pivoting the PostgreSQL Way

PostgreSQL does not have a native PIVOT keyword. The common pattern is conditional aggregation with the FILTER clause. It is readable, explicit, and easy to extend.

SELECT
    DATE_TRUNC('month', order_date) AS sales_month,
    SUM(total_amount) FILTER (WHERE order_status = 'paid') AS paid_amount,
    SUM(total_amount) FILTER (WHERE order_status = 'shipped') AS shipped_amount,
    SUM(total_amount) FILTER (WHERE order_status = 'cancelled') AS cancelled_amount
FROM sales.orders
GROUP BY DATE_TRUNC('month', order_date)
ORDER BY sales_month;

Module 5: Stored Procedures & Temporary Tables

PostgreSQL procedures were introduced in version 11. Unlike functions, procedures can manage transactions with COMMIT and ROLLBACK when executed in an appropriate call context. That makes them useful for controlled batch workflows.

CREATE OR REPLACE PROCEDURE billing.process_batch_payments(
    IN p_batch_id BIGINT
)
LANGUAGE plpgsql
AS $$
BEGIN
    CREATE TEMP TABLE payment_worklist (
        order_id BIGINT,
        user_id BIGINT,
        amount NUMERIC(12, 2)
    ) ON COMMIT DROP;

    INSERT INTO payment_worklist (order_id, user_id, amount)
    SELECT order_id, user_id, total_amount
    FROM sales.orders
    WHERE payment_batch_id = p_batch_id
      AND order_status = 'pending_payment';

    UPDATE sales.orders AS o
    SET order_status = 'paid',
        paid_at = NOW()
    FROM payment_worklist AS pw
    WHERE o.order_id = pw.order_id;

    COMMIT;
EXCEPTION
    WHEN OTHERS THEN
        ROLLBACK;
        RAISE;
END;
$$;

ON COMMIT DROP guarantees cleanup. That is useful when temp data should never survive the transaction boundary, especially in batch-style procedures.

Module 6: User-Defined Functions (UDFs)

Scalar Function

A scalar function returns one value. Use it when the logic is small, deterministic, and genuinely reusable.

CREATE OR REPLACE FUNCTION billing.calculate_order_tax(
    p_subtotal NUMERIC,
    p_tax_rate NUMERIC
)
RETURNS NUMERIC
LANGUAGE plpgsql
AS $$
BEGIN
    RETURN ROUND(p_subtotal * p_tax_rate, 2);
END;
$$;
SELECT billing.calculate_order_tax(249.99, 0.0825) AS tax_amount;

Table-Returning Function

A table function returns a result set. It is useful when you want reusable parameterized query logic that behaves like a virtual table.

CREATE OR REPLACE FUNCTION reporting.get_user_orders(
    p_user_id BIGINT
)
RETURNS TABLE (
    order_id BIGINT,
    order_date TIMESTAMP WITH TIME ZONE,
    order_status TEXT,
    total_amount NUMERIC(12, 2)
)
LANGUAGE plpgsql
AS $$
BEGIN
    RETURN QUERY
    SELECT
        o.order_id,
        o.order_date,
        o.order_status,
        o.total_amount
    FROM sales.orders AS o
    WHERE o.user_id = p_user_id
    ORDER BY o.order_date DESC;
END;
$$;
SELECT *
FROM reporting.get_user_orders(42);

Module 7: Bonus - Advanced Data Types (JSONB)

PostgreSQL is famous for JSON support because it lets you mix relational structure with flexible semi-structured payloads. That is useful when the core entity is stable but some attributes vary, such as marketplace-specific metadata, checkout payloads, or webhook responses.

CREATE TABLE sales.order_events (
    event_id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    order_id BIGINT NOT NULL,
    event_type TEXT NOT NULL,
    event_payload JSONB NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
);
SELECT
    order_id,
    event_payload ->> 'source' AS source_system,
    event_payload -> 'payment' ->> 'provider' AS payment_provider
FROM sales.order_events
WHERE event_type = 'checkout_completed';
SELECT order_id, event_payload
FROM sales.order_events
WHERE event_payload @> '{"payment": {"provider": "stripe"}}'::jsonb;
💡
Design guideline: keep highly queried, strongly typed business fields in regular columns. Use JSONB for flexible attributes, payload snapshots, or edge-case metadata, not as an excuse to avoid data modeling.
